# Поэтапный план реализации

## Этап 1: Проектирование архитектуры и спецификаций

| Параметр | Значение |
|----------|----------|
| **Роль** | System Architect / Tech Lead |
| **Контекст** | Начало проекта. Необходимо заложить фундамент для масштабируемой системы анализа текстов с поддержкой 10+ конкурентов и 500+ ключевых слов |
| **Что сделано** | - Определены бизнес-требования к отчёту<br>- Создана черновая JSON-схема данных |
| **Что сейчас делаем** | Детализируем архитектуру, утверждаем контракты API и форматы данных |

### Пошаговый план

1. Провести воркшоп с аналитиками и SEO-специалистами для валидации метрик
2. Спроектировать ER-диаграмму для хранения результатов анализа
3. Утвердить JSON Schema для валидации отчётов
4. Определить интерфейсы сервисов (Parser, Analyzer, Reporter)
5. Задокументировать в OpenAPI/Swagger endpoints для API

### Технологии

- **Документация**: Swagger/OpenAPI 3.0, Mermaid для диаграмм
- **Валидация**: JSON Schema Draft 7
- **Хранение**: PostgreSQL (реляционные данные), MongoDB (гибкие отчёты)
- **Коммуникация**: AsyncAPI для event-driven архитектуры

### Проверка после этапа

- [ ] Все стейкхолдеры подписали спецификацию
- [ ] JSON Schema валидирует тестовые данные из knowledge base
- [ ] ER-диаграмма покрывает все сущности отчёта
- [ ] OpenAPI spec проходит валидацию через `swagger-cli`

## Этап 2: Сбор и парсинг данных с сайтов

| Параметр | Значение |
|----------|----------|
| **Роль** | Backend Developer / Data Engineer |
| **Контекст** | Архитектура утверждена. Необходимо реализовать надёжный парсер для 11+ сайтов с защитой от блокировок |
| **Что сделано** | - Спецификации данных согласованы<br>- Выбраны технологии для парсинга |
| **Что сейчас делаем** | Реализуем модуль сбора контента с конкурентных сайтов |

### Пошаговый план

1. Настроить HTTP-клиент с ротацией User-Agent и прокси
2. Реализовать парсер HTML → извлечение текста, заголовков, meta-тегов
3. Добавить обработку динамического контента (Selenium/Playwright при необходимости)
4. Реализовать кэширование ответов для повторных запусков
5. Добавить логирование и мониторинг успешности парсинга
6. Создать адаптеры под разные структуры сайтов (Tripster, LaRest, Sputnik8 и др.)

### Технологии

- **Парсинг**: BeautifulSoup4, lxml, Selectolax (для скорости)
- **HTTP**: httpx с asyncio, retry-логика через tenacity
- **Динамический контент**: Playwright (легковеснее Selenium)
- **Кэширование**: Redis с TTL 24h
- **Очереди**: Celery + RabbitMQ для асинхронной обработки
- **Мониторинг**: Prometheus metrics + Sentry для ошибок

### Проверка после этапа

- [ ] Парсер успешно извлекает данные со всех 11 тестовых URL
- [ ] Время парсинга одного сайта < 15 секунд
- [ ] Кэш снижает нагрузку на повторных запросах на 90%
- [ ] Обработаны кейсы: 404, 403, таймауты, капча
- [ ] Логирование позволяет отладить любой неудачный парсинг

## Этап 3: Лемматизация и анализ текста

| Параметр | Значение |
|----------|----------|
| **Роль** | NLP Engineer / Python Developer |
| **Контекст** | Данные собраны. Необходимо нормализовать текст и рассчитать метрики качества |
| **Что сделано** | - Контент сайтов сохранён в БД<br>- Настроены пайплайны обработки |
| **Что сейчас делаем** | Реализуем модуль лингвистического анализа: лемматизация, расчёт тошноты, водянистости, закона Ципфа |

### Пошаговый план

1. Подключить pymorphy3 для лемматизации русского языка
2. Реализовать токенизацию с учётом стоп-слов и морфологии
3. Рассчитать метрики:
   - `Тошнота` = (частота самого частого слова / общее число слов) × 100
   - `Водянистость` = доля местоимений/предлогов/союзов
   - `Закон Ципфа` = ранг слова × частота, сравнение с идеальной кривой
4. Построить частотные словари для биграмм/триграмм (nltk.collocations)
5. Рассчитать плотность: вхождения / общее число слов × 100%
6. Экспортировать результаты в промежуточный формат для отчёта

### Технологии

- **Лемматизация**: pymorphy3 (быстрее pymorphy2)
- **Токенизация**: natasha + razdel для русского языка
- **N-граммы**: nltk.collocations, own implementation для производительности
- **Математика**: numpy для расчёта Ципфа, scipy для статистики
- **Оптимизация**: Cython для критичных участков, multiprocessing для параллелизации

### Проверка после этапа

- [ ] Лемматизация корректно обрабатывает формы: "экскурсии" → "экскурсия"
- [ ] Метрика "тошнота" для тестового текста совпадает с эталоном ±0.5
- [ ] Биграммы типа "пятигорск экскурсия" детектируются с плотностью 0.94%
- [ ] Обработка текста 10 000 слов занимает < 3 секунд
- [ ] Результаты анализа сохраняются в БД и доступны для генерации отчёта

## Этап 4: Сравнительный анализ и расчёт рекомендаций

| Параметр | Значение |
|----------|----------|
| **Роль** | Data Analyst / SEO Specialist (в паре с разработчиком) |
| **Контекст** | Метрики рассчитаны для всех страниц. Необходимо сравнить целевую страницу с конкурентами и сформировать рекомендации |
| **Что сделано** | - Все тексты проанализированы и нормализованы<br>- Метрики сохранены в едином формате |
| **Что сейчас делаем** | Реализуем алгоритмы сравнения и генерации приоритизированных рекомендаций |

### Пошаговый план

1. Рассчитать медиану и среднее по каждому показателю среди конкурентов
2. Для каждого ключевого слова вычислить:
   - Разницу между целевой страницей и медианой
   - Рекомендуемое количество вхождений (медиана ± допустимое отклонение)
   - Приоритет оптимизации: критический если |разница| > 50%
3. Реализовать эвристики для мета-тегов:
   - Title: длина 50-60 символов, ключ в начале
   - Description: 150-160 символов, уникальность
4. Сгенерировать рекомендации с группировкой по приоритету
5. Рассчитать итоговые скоры (0-100) для каждого блока отчёта

### Технологии

- **Статистика**: pandas для агрегации, numpy для расчётов
- **Правила**: JSON-based rule engine (json-rules-engine) для гибкости
- **Приоритизация**: взвешенная формула: `priority = impact × effort⁻¹ × confidence`
- **Генерация текста**: Jinja2 для шаблонов рекомендаций
- **Тестирование**: pytest с фикстурами на основе реальных данных из knowledge base

### Проверка после этапа

- [ ] Для слова "экскурсия" разница -16 корректно интерпретируется как "недостаточно"
- [ ] Рекомендации для "тошнота=8 при медиане=7" имеют приоритет medium, не critical
- [ ] Meta-тег с Title=120 символов получает рекомендацию "сократить до 60"
- [ ] Итоговый SEO-score для тестовой страницы воспроизводим ±2 балла
- [ ] Все рекомендации содержат конкретные действия, а не общие фразы

## Этап 5: Генерация отчёта и экспорт

| Параметр | Значение |
|----------|----------|
| **Роль** | Fullstack Developer / DevOps |
| **Контекст** | Данные анализа и рекомендации готовы. Необходимо собрать финальный отчёт в утверждённом формате |
| **Что сделано** | - Все расчёты выполнены и протестированы<br>- Рекомендации сгенерированы |
| **Что сейчас делаем** | Собираем финальный JSON-отчёт, добавляем валидацию и экспорт в разные форматы |

### Пошаговый план

1. Создать сервис-композитор, который собирает данные из всех модулей
2. Валидировать итоговый объект по JSON Schema из Этапа 1
3. Реализовать экспорт в форматы: JSON (основной), CSV (для Excel), PDF (для клиентов)
4. Добавить сжатие и подписывание отчётов (опционально)
5. Настроить endpoint API: `POST /api/v1/analyze` → `202 Accepted` + webhook / polling
6. Реализовать кэширование готовых отчётов (TTL 1 час)

### Технологии

- **Сборка отчёта**: Pydantic для типизации и валидации данных
- **Экспорт**:
  - JSON: стандартная библиотека
  - CSV: pandas.to_csv с кастомными форматерами
  - PDF: WeasyPrint или ReportLab
- **API**: FastAPI с async support, background tasks для долгих расчётов
- **Кэширование**: Redis + hashlib для key generation по параметрам запроса
- **Безопасность**: JWT для аутентификации, rate limiting через slowapi

### Проверка после этапа

- [ ] Отчёт валидируется по JSON Schema без ошибок
- [ ] Экспорт в CSV сохраняет все числовые значения без потерь точности
- [ ] Генерация полного отчёта (11 сайтов, 500 ключей) < 30 секунд
- [ ] API возвращает корректные статусы: 202 для async, 400 при невалидных входных данных
- [ ] Повторный запрос с теми же параметрами возвращается из кэша < 100 мс

## Этап 6: Тестирование, документирование и деплой

| Параметр | Значение |
|----------|----------|
| **Роль** | QA Engineer / Technical Writer / DevOps |
| **Контекст** | Система функциональна. Необходимо обеспечить надёжность, понятность и готовность к продакшену |
| **Что сделано** | - Все модули реализованы и интегрированы<br>- Отчёты генерируются корректно |
| **Что сейчас делаем** | Проводим комплексное тестирование, пишем документацию, настраиваем CI/CD |

### Пошаговый план

1. Написать интеграционные тесты на полный пайплайн (парсинг → анализ → отчёт)
2. Провести нагрузочное тестирование: 100 одновременных запросов
3. Создать пользовательскую документацию:
   - API reference (автогенерация из OpenAPI)
   - Guide по интерпретации метрик отчёта
   - FAQ по частым вопросам оптимизации
4. Настроить CI/CD пайплайн:
   - Линтеры: black, flake8, mypy
   - Тесты: pytest с coverage > 85%
   - Сборка: Docker multi-stage
   - Деплой: Kubernetes Helm chart или Docker Compose для staging
5. Настроить мониторинг: health checks, логирование запросов, алерты на ошибки

### Технологии

- **Тестирование**: pytest + pytest-asyncio, locust для нагрузочного, hypothesis для property-based
- **Документация**: MkDocs + mkdocstrings для автогенерации, Swagger UI для API
- **CI/CD**: GitHub Actions или GitLab CI, Docker, Trivy для сканирования уязвимостей
- **Деплой**: Docker + docker-compose (MVP), Kubernetes (scale), Terraform для инфраструктуры
- **Мониторинг**: Prometheus + Grafana дашборды, ELK stack для логов

### Проверка после этапа

- [ ] Интеграционный тест проходит на реальных данных из knowledge base
- [ ] Нагрузка 100 RPS не вызывает ошибок > 1% и latency > 2s
- [ ] Покрытие кода тестами ≥ 85%, критические пути — 100%
- [ ] Документация позволяет новому разработчику запустить проект за < 1 час
- [ ] Деплой в staging происходит автоматически при мердже в main
- [ ] Алерт срабатывает при падении успешности парсинга ниже 95%
