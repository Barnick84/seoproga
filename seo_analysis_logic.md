# Реализация SEO Анализа и Истории Оптимизации

В данном документе подробно описана логика и архитектура работы модуля **SEO-анализа страниц и ведения истории оптимизации**.

---

## Архитектура системы

Реализация разделена на четыре уровня:
1. **База данных (MySQL)**: Хранение истории отчетов для каждого кластера по датам.
2. **Бэкенд (Node.js/Express)**: API-эндпоинты для вызова парсинга, сохранения в БД и раздачи данных фронтенду.
3. **Python-скрипт (`generate_seo_plan.py`)**: Сбор данных, умный парсинг HTML-кода, логика разделения контекста и интеграция с LLM (Gemini 3.1 Pro / OpenAI).
4. **Интерфейс (HTML/JS)**: Вкладка "SEO анализ" в панели управления кластером с чекбоксом выбора режима оптимизации и выпадающим списком истории.

---

## 1. Логика Сбора Данных и Интентов (Python)

Скрипт `scripts/generate_seo_plan.py` запускается бэкендом, считывает метрики последнего общего анализа конкурентов по `cluster_id` и выбирает стратегию в зависимости от **интента**:

### А. Коммерческий интент (Страница услуг / товара)
Для коммерческих страниц важно оптимизировать только контентную часть, не ломая общую структуру сайта (шапка, подвал, сайдбар, списки товаров).
1. **Изоляция контента с помощью BeautifulSoup4**:
   - Скрипт ищет тег `<article>` (наиболее частый семантический контейнер статьи или основного текста).
   - Если `<article>` не найден, происходит fallback на тег `<main>`.
   - Если нет ни `<article>`, ни `<main>`, скрипт завершает работу с ошибкой и возвращает пользователю подробную инструкцию о необходимости внедрения семантической верстки.
2. **Разделение контента**:
   - Найденный тег вырезается из HTML.
   - Скрипт разделяет исходную страницу на три части: `before_content` (все, что идет до тега), `target_content` (внутренности тега для оптимизации), и `after_content` (все, что идет после).
3. **Формирование контекстного промпта для LLM**:
   - Нейросети отправляются все три части, чтобы она понимала глобальный контекст страницы, дизайн-систему и стиль.
   - Нейросети отправляются медианные показатели конкурентов (объем текста в словах, уровень тошноты, водность, ключевые LSI фразы).
   - **Управление режимом («Переписать полностью» vs «Дооптимизировать»)**:
     - Если передан флаг `rewrite_content = True`, LLM получает команду: *"ПОЛНОСТЬЮ ПЕРЕПИШИТЕ контент внутри целевого тега с нуля, ориентируясь на метрики"*.
     - Если `rewrite_content = False`, LLM получает команду: *"ДО-ОПТИМИЗИРУЙТЕ существующий контент внутри целевого тега, сохранив текущий смысл, но улучшив вхождения ключей и SEO-метрики"*.
   - Дается жесткая инструкция: **не изменять** код блоков `before_content` и `after_content`.

### Б. Информационный интент (Статья / Блог)
Для информационных запросов целью является создание структуры будущей статьи:
1. **Извлечение структуры заголовков**: Скрипт парсит текущие заголовки страницы (H1-H6).
2. **Генерация структуры**: На основе структуры конкурентов и медианных SEO-показателей LLM создает идеальную H1-H3 иерархию для будущей статьи.
3. **Генерация ТЗ**: LLM генерирует подробный промпт (техническое задание) для копирайтера, включающий: медианный объем статьи, разрешенные показатели спамности и список LSI-ключей, которые необходимо употребить в каждом из разделов.

## 2. Логика Хранения (База Данных)

Создана таблица `cluster_seo_history` со следующей структурой:
```sql
CREATE TABLE IF NOT EXISTS cluster_seo_history (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    site_url VARCHAR(255) NOT NULL,
    cluster_id INT NOT NULL,
    analysis_date DATE NOT NULL,
    intent_type VARCHAR(50) NOT NULL,
    seo_plan_content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY user_site_cluster_date (user_id, site_url, cluster_id, analysis_date)
);
```

**Особенности работы**:
- Поле `UNIQUE KEY` по связке `(user_id, site_url, cluster_id, analysis_date)` предотвращает создание дубликатов на одну и ту же дату.
- При повторном запуске генерации в тот же день срабатывает инструкция `ON DUPLICATE KEY UPDATE`, которая обновляет уже существующую запись за сегодня.

## 3. Бэкенд Интеграция (Node.js)

В `server.js` добавлены эндпоинты:
- **`GET /api/seo-history/dates`**: Возвращает массив дат (`YYYY-MM-DD`), на которые для данного кластера уже сохранены планы оптимизации. Даты сортируются от новых к старым.
- **`GET /api/seo-history/plan`**: Принимает `date` и отдает сохраненный Markdown-текст плана и интент.
- **`POST /api/seo-history/generate`**:
  - Принимает `domain`, `clusterId` и `rewriteContent`.
  - Запускает Python-скрипт с помощью `child_process.spawn`.
  - Передает параметры в виде аргументов командной строки: `python.exe generate_seo_plan.py <domain> <clusterId> <userId> <1 или 0>`.
  - После завершения возвращает JSON с результатом.

## 4. Клиентский Интерфейс (HTML/JS)

1. **Новая вкладка**: В `public/analysis.html` добавлена вкладка "SEO анализ" рядом с "Мета-теги".
2. **Чекбокс**: Стилизован чистым CSS кроссбраузерный чекбокс `<input type="checkbox" id="rewriteContentCheckbox">` («Переписать контент»). Он имеет плавный hover-эффект, кастомный индикатор (checkmark) и гармонично вписан в общую палитру интерфейса.
3. **Markdown-рендер**: Подключена внешняя библиотека `marked.js` с CDN, которая форматирует возвращаемый LLM Markdown в красивый HTML с заголовками, списками и кодом.
4. **Умное поведение dropdown**:
   - При первой загрузке страницы JS автоматически запрашивает список дат через `/api/seo-history/dates`.
   - Если отчеты найдены, в выпадающий список подгружаются даты, и по умолчанию рендерится самый свежий (последний) отчет.
   - Если отчетов нет, выводится заглушка с предложением сгенерировать первый план.
